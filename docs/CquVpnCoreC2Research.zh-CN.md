# CquVpnCore C2 研究记录：终端注册门槛

状态：未通过 C2。本文记录的是研究结论和继续条件，不是 VPN 使用说明，也不提供任何连接参数。

## 研究范围

目标是判断 `CquVpnCore` 能否在完全不使用既有官方客户端材料的前提下，为一台新终端完成服务器认可的注册。只有这一点得到实证，才有资格研究后续资源协商和传输层。

研究明确排除了密码、验证码、Cookie、浏览器存储、会话票据、终端标识、证书、私钥、配置文件和厂商二进制的读取、导出、重放或复用。

## 已确认的事实

1. 浏览器完成统一身份认证不等于已建立 VPN。桌面层的可见调用顺序是浏览器认证、取得一次性会话材料、再交给本地代理发起后续连接；其中最后两步的服务器契约并没有对独立客户端公开。
2. 已解包的桌面层能够访问的是本地 Thrift 状态、事件和诊断接口。可见方法用于读取状态、订阅事件、退出和诊断，没有发现面向新终端的注册接口，也没有发现可由第三方独立调用的建隧道 RPC。
3. SPA、UDP 敲门和 TLS 代理行为位于原生组件中，而不是标准的 WireGuard、OpenVPN 或 Clash 配置层。仅有浏览器认证结果无法推导出可用的通用客户端配置。
4. 对公开代码索引中关键调用名的查询没有发现可审计的开源协议实现、CLI、SDK 或终端注册说明。这个结果只能说明“当前未找到已验证的公开路径”，不能证明任何接口绝对不存在。
5. 当前设备上的官方服务处于停止且禁用状态。CquAutoLogin 的 C0/C1 实现不会启动该服务、Agent 或 Tray。

## 当前结论

C2 的必要证据尚不存在：没有证明一把由 `CquVpnCore` 新建、且不导入厂商材料的终端身份会被服务器接受。因此当前版本不会实现注册请求、隧道协商、虚拟网卡、路由、DNS、系统代理或 Clash 配置导出，也不能宣称用户可以卸载官方客户端后继续使用 VPN。

这不是把问题归结为“浏览器认证失败”。浏览器认证检测已经属于 C1；缺失的是服务器侧对全新终端身份和后续资源协商的可验证支持。

## 继续 C2 所需证据

以下任一路径都必须在隔离环境中，用新建身份和无敏感导出的记录完成验证：

1. 获得网络服务提供方公开或书面认可的终端注册契约，包括请求边界、身份绑定规则和测试环境。
2. 在获得授权的测试账户与测试控制器上，由 CquVpnCore 新生成不可导出的 Windows CNG 密钥，并确认服务器接受该新身份。
3. 获得一个由服务端提供的、面向第三方客户端的标准协议入口；只有经实测确认后，才评估标准客户端兼容性。

在上述任一证据出现前，下一阶段只允许改善 C0/C1 的可观测性、浏览器认证体验和失败提示，不允许猜测协议或改动系统网络配置。

## 验收标准

进入 C2 的唯一验收信号是：在一台干净 Windows 环境中，官方客户端未运行且不被读取，CquVpnCore 使用自身生成的新身份完成一次服务端认可的注册；记录只保留非敏感的结果类别、时间和操作标识。失败时必须没有遗留的网卡、路由、DNS、防火墙或服务改动。
